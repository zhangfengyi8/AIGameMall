# 王者荣耀导购 Agent Skills 设计

## 目标

在仓库内提供两个运行时 skill 资产，供 `services/agent-orchestrator` 后续加载或转写为 Agent prompt/workflow。两个 skill 面向王者荣耀账号交易导购：

- `buyer-requirement-intake`：拆解用户买号需求，并在需求模糊时引导用户补全关键条件。
- `account-recommendation-brief`：把已澄清需求转成账号筛选、排序、推荐理由和风险提示策略。

本设计只落地可读、可复用的 skill 说明文件，不实现 Agent 编排代码、库存查询代码或前端页面。

## 目录结构

```text
services/agent-orchestrator/skills/
  buyer-requirement-intake/
    SKILL.md
  account-recommendation-brief/
    SKILL.md
```

每个目录只包含一个 `SKILL.md`。后续如果导购系统需要程序化加载，可在当前结构上增加 manifest 或 prompt loader。

## Skill 1: buyer-requirement-intake

职责：

- 判断用户是否在表达王者荣耀账号购买、比较、找号或咨询需求。
- 从自然语言中抽取可执行的买号条件。
- 判断需求是否足够进入推荐阶段。
- 当需求模糊时，优先问最少、最关键的问题，避免一次性问太多。
- 保持交易合规和风险意识，不承诺绕过实名、防找回或平台规则。

核心槽位：

- `game`：默认王者荣耀。
- `platform`：QQ、微信、安卓、iOS、跨系统需求。
- `server_region`：区服、大区、同区诉求。
- `budget`：预算区间、最高价、是否可浮动。
- `account_goal`：收藏皮肤、排位上分、战力号、英雄池、贵族等级、性价比、送礼等。
- `rank`：当前段位、历史段位、巅峰赛分数。
- `heroes`：指定英雄、常用分路、英雄数量。
- `skins`：指定皮肤、皮肤品质、限定/典藏/无双/传说/史诗、皮肤数量。
- `assets`：铭文、贵族等级、荣耀水晶、点券、金币、改名卡等。
- `risk_preference`：实名认证、二次实名、找回风险、是否只看平台担保。
- `deal_preference`：价格优先、资产优先、同区优先、可接受缺项。

输出建议：

```json
{
  "intent": "buy_account",
  "confidence": 0.0,
  "slots": {},
  "missing_required_slots": [],
  "missing_optional_slots": [],
  "clarifying_question": "",
  "ready_for_recommendation": false,
  "notes": []
}
```

进入推荐阶段的最低条件：

- 用户明确是买号/找号/比较账号。
- 至少有预算或核心目标之一。
- 平台/系统不明确时，如果库存支持跨平台搜索，可先推荐并提示；如果库存强依赖平台，应优先澄清。

澄清策略：

- 一轮只问 1 到 2 个最影响召回的问题。
- 优先问预算和平台/系统；其次问核心皮肤、英雄、段位或账号用途。
- 对“随便推荐”“性价比高”“毕业号”等模糊词，转成可选方向让用户选。
- 已有足够信息时不要继续盘问，直接进入推荐阶段。

## Skill 2: account-recommendation-brief

职责：

- 接收 intake 输出或等价的已澄清需求。
- 区分硬筛条件和软偏好。
- 生成账号查询条件、排序权重、推荐解释和风险提示。
- 在无完全匹配时给出降级策略，例如放宽预算、皮肤数量、区服或段位。
- 避免虚构库存数据；只基于调用方提供的候选账号或查询工具结果推荐。

硬筛条件示例：

- 平台/系统不可错。
- 用户指定的必选皮肤或英雄不可缺。
- 最高预算不可超，除非用户明确可浮动。
- 用户明确拒绝高风险账号时，过滤实名异常、找回风险高或交易保障不足的账号。

软偏好示例：

- 皮肤品质和稀有度。
- 常用英雄覆盖度。
- 段位、巅峰分、战力。
- 铭文、贵族等级、荣耀水晶等资产。
- 价格接近预算下限或性价比高。

输出建议：

```json
{
  "query": {
    "filters": {},
    "must_have": [],
    "nice_to_have": []
  },
  "ranking": {
    "weights": {},
    "tie_breakers": []
  },
  "recommendation_policy": {
    "max_items": 3,
    "explain_fields": [],
    "risk_checks": [],
    "fallbacks": []
  },
  "buyer_message_guidance": []
}
```

推荐解释要求：

- 先说匹配用户最关心的点，再说价格和风险。
- 每个账号解释 2 到 4 个要点即可。
- 对缺项要明说，例如“没有指定的某款皮肤，但预算内传说皮肤更多”。
- 风险提示要具体，不使用恐吓式话术。

## 数据关系

当前仓库只有 `data/skins.json` 和字段说明，可用于识别英雄、皮肤名、品质和标签。账号库存 schema 尚未存在，因此两个 skill 暂不绑定具体账号字段名，只定义可迁移的语义槽位和输出契约。

后续新增账号数据时，应补充：

- 账号商品字段映射。
- 必选/可选槽位到查询字段的映射。
- 排序权重的默认数值。
- 风险字段的枚举定义。

## 验证

人工验证以下场景：

- “预算 500，想买个李白凤求凰号”应提取预算、英雄、皮肤，并询问平台/系统或直接标记可推荐。
- “帮我找个性价比高的号”应优先询问预算和平台/系统。
- “安卓 QQ，1000 内，皮肤多点，最好有几个传说”应进入推荐阶段，并把皮肤数量/传说皮肤作为软偏好。
- “有没有不会找回的号”应提示只能筛选低风险和担保交易，不承诺绝对不会找回。
- 无候选账号时，推荐 skill 应输出降级策略，而不是编造账号。
